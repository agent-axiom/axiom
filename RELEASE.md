# AXIOM Release Verification

AXIOM release transparency is built around GitHub Releases and GitHub Actions.

Each tagged release is expected to publish:
- wheel
- source distribution
- `SHA256SUMS`
- `SBOM.spdx.json`

The release workflow also generates GitHub artifact attestations for the release assets and the SBOM.

## What Users Can Verify

Users can verify:
- which AXIOM version is installed locally
- which commit and tag a release build came from
- that a downloaded release asset matches the published `SHA256SUMS`
- that a release asset has a valid GitHub attestation

Users cannot assume:
- native binary transparency, because AXIOM is currently source-first
- reproducible builds, because this slice does not implement that yet
- immutable release guarantees unless the repository has immutable releases enabled in GitHub settings

## Inspect Local Build Metadata

Run:

```bash
bin/axiom version --verbose
```

Expected fields:
- `version`
- `git_commit`
- `git_tag`
- `build_timestamp`
- `source_repo`

## Verify Release Asset Hashes

Download the release assets, then compare them to `SHA256SUMS`.

Example:

```bash
shasum -a 256 axiom_workflow-0.1.0-py3-none-any.whl
shasum -a 256 axiom_workflow-0.1.0.tar.gz
cat SHA256SUMS
```

The digest for each file must match the corresponding line in `SHA256SUMS`.

## Verify GitHub Attestations

With GitHub CLI installed, verify an asset directly against the repo attestations:

```bash
gh attestation verify ./axiom_workflow-0.1.0-py3-none-any.whl \
  --repo agent-axiom/axiom \
  --signer-workflow agent-axiom/axiom/.github/workflows/release.yml \
  --source-ref refs/tags/v0.1.0
```

This verifies provenance for the local artifact against the attestation generated by the release workflow.

## Verify GitHub Release Integrity

If the repository uses GitHub immutable releases, you can also verify the
release and local asset through the release attestation flow described in GitHub
docs:

```bash
gh release verify v0.1.0 --repo agent-axiom/axiom
gh release verify-asset v0.1.0 ./axiom_workflow-0.1.0-py3-none-any.whl --repo agent-axiom/axiom
```

Whether this is available depends on repository settings and GitHub release features, not just the AXIOM workflow.

## Closed Infrastructure Promotion Flow

For a closed environment, the recommended flow is:

1. Download release assets from GitHub in a connected environment.
2. Verify `SHA256SUMS`.
3. Verify GitHub artifact attestations.
4. Mirror the verified assets into the internal package or binary store.
5. Promote only the verified assets into developer or production environments.

That keeps the trust boundary at the promotion step rather than at ad hoc installation time.
